> 生成时间：2026-04-01
> 状态：规划中

## 核心定位

```
一句话：在三维世界的墙上写字、贴照片。

产品形态：多人公共元宇宙城市
         每个人可以建造自己的建筑
         别人可以来逛，但只有主人能编辑
```

## 数据库设计

### 数据层次

```
┌─────────────────────────────────────────────────────────────┐
│                        数据层次                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   City（城市）                                              │
│     └── Building（建筑）- 全局唯一名称，像域名               │
│           └── Floor（楼层）- 可选，大型建筑需要              │
│                 └── Board（黑板）- 最多 300 个/建筑          │
│                       └── Content（内容）- text/image/canvas│
│                                                             │
│   User（用户）                                              │
│     └── 拥有的建筑列表                                       │
│     └── 访问权限（哪些是只读、哪些可编辑）                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 数据表设计

```sql
-- 用户表（V3.0）
CREATE TABLE users (
  id TEXT PRIMARY KEY,           -- UUID 或用户名
  email TEXT,
  display_name TEXT,
  created_at DATETIME
);

-- 建筑表
CREATE TABLE buildings (
  id TEXT PRIMARY KEY,           -- 全局唯一，如"我的小屋"
  owner_id TEXT,                 -- 外键 → users.id
  display_name TEXT,
  description TEXT,
  is_public BOOLEAN DEFAULT 1,   -- 是否公开可逛
  visitor_count INTEGER DEFAULT 0,
  created_at DATETIME
);

-- 黑板表
CREATE TABLE boards (
  id INTEGER PRIMARY KEY,        -- 全局自增 ID
  building_id TEXT NOT NULL,
  local_index INTEGER,           -- 建筑内的序号：0, 1, 2...
  name TEXT,
  mode TEXT DEFAULT 'text',      -- text/image/canvas
  content TEXT,
  created_at DATETIME,
  updated_at DATETIME,
  FOREIGN KEY (building_id) REFERENCES buildings(id)
);

-- Canvas 函数库（全局共享）
CREATE TABLE canvas_functions (
  name TEXT PRIMARY KEY,
  code TEXT,
  created_at DATETIME
);
```

### 索引策略

```sql
-- 高频查询优化
CREATE INDEX idx_boards_building ON boards(building_id);
CREATE INDEX idx_buildings_owner ON buildings(owner_id);
CREATE INDEX idx_buildings_public ON buildings(is_public);

-- 未来：全文搜索

## V1.0 ~ V10.0 路线图

### V1.0 「城市原型」— 2-3 天发布

```
目标：验证核心价值「在三维墙上写字」

功能：
  ✅ 单用户（你）
  ✅ 3 个建筑（手动创建）
  ✅ 每建筑 5 黑板
  ✅ text/image/canvas 三种模式
  ✅ SQLite 本地存储
  ✅ admin.html 分组编辑
  ✅ 三维里实时看到更新

数据：
  - 15 条黑板记录
  - 无用户系统

发布方式：
  - 本地运行（不给别人）
  - 发一个技术演示视频
  - 收集反馈，不追求完美

发布渠道：
  - 知乎文章 + 演示视频
  - GitHub README 更新
  - 朋友圈/微信群分享
```

### V2.0 「建筑自由」— 2 周

```
目标：让用户能自己创建建筑

新增：
  ✅ 建筑创建/删除
  ✅ 建筑描述编辑
  ✅ 黑板数量扩展（最多 50 个/建筑）
  ✅ admin 建筑管理页面

数据：
  - 用户可自由添加建筑
  - 仍然单用户

盈利：无（积累种子用户）

目标：真正的多人元宇宙

  ✅ 用户注册/登录

## 盈利模式

### 不推荐的路径

```
❌ 卖给企业做效率工具
   → 产品不是效率工具，是「慢空间」

❌ 免费做大，以后再说
   → 服务器成本会拖垮你

❌ 广告
   → 破坏沉浸感，违背产品哲学
```

### 推荐的路径

```
✅ 订阅制 + 装饰商城 混合

免费层：
  - 3 个建筑
  - 每建筑 50 黑板
  - 基础装饰

Pro（$5/月）：
  - 无限建筑
  - 每建筑 300 黑板
  - 高级装饰
  - AI 整理助手
  - 导出功能

装饰商城：
  - 建筑皮肤：$1-5
  - 黑板边框：$0.5-2
  - 特效：$2-10
  - 创作者可上架分成
```

### 收入预测（乐观）

```
假设 1 年后：
  - 1000 注册用户
  - 5% 转化率 → 50 付费用户
  - $5/月 × 50 × 12 = $3000/年

装饰商城：
  - 10% 用户购买 → 100 用户
  - 平均 $10/年 → $1000/年

总计：$4000/年

这不算多，但可以覆盖服务器成本。
真正的收入来自 V6.0+ 的规模化。
```

## 风险点

```
1. 用户习惯迁移成本高
   → 大多数人用惯了平面笔记
   → 应对：定位为「补充」而非「替代」

2. 服务器成本（多人后）
   → 三维渲染在前端，服务器成本可控
   → 应对：SQLite → PostgreSQL → 分库分表

3. 被大厂抄袭
   → 概念可以抄，40kb 引擎抄不走
   → 应对：持续迭代，建立社区
```

## 深度讨论要点（2026-04-01 追加）

> 超过 V1.0 的规划都是幻想，以下是 V1.0 的关键决策点。

### 1. 产品定位

```
核心价值：在三维世界的墙上写字、贴照片

产品形态：
  - 多人公共元宇宙城市
  - 城东0034号 = 考研记忆馆
  - 城西098号 = 历史展览
  - 城南066号 = 我的小屋
  - 任何人可以逛，主人才能编辑
```

### 2. 三维空间 vs admin 平面

```
三维空间 = 核心产品（用户来是为了三维体验）
admin    = 辅助工具（三维空间的编辑后台）

admin 不是「另一个笔记软件」
admin 是「三维空间的 CMS」
```

### 3. 数据约束

```
每个建筑最多 300 个画板（硬性限制）
  → 制造稀缺性，用户会认真考虑放什么
  → 300 个已是「大房间」
  → 数量有限 = 加载快，体验好
```

### 4. 数据库设计（V1.0 最小版）

```sql
-- 建筑（全局唯一名称，像域名）
CREATE TABLE buildings (
  name TEXT PRIMARY KEY,    -- "我的小屋"、"考研记忆馆"
  description TEXT
);

-- 画板（建筑内的序号，0-299）
CREATE TABLE boards (
  building_name TEXT NOT NULL,
  local_id INTEGER NOT NULL,   -- 0, 1, 2... 299
  mode TEXT DEFAULT 'text',    -- text/image/canvas
  content TEXT,
  PRIMARY KEY (building_name, local_id),
  FOREIGN KEY (building_name) REFERENCES buildings(name),
  CHECK (local_id >= 0 AND local_id < 300)
);
```

**关键设计决策**：
- 画板主键是 `(building_name, local_id)`，不是全局自增 ID
- 这样设计的好处：导出成文件时 → `/建筑名/local_id.md`
- 为未来迁移到文件存储预留空间

### 5. admin 界面结构

```
┌─────────────────────────────────────────────────────────────┐
│  admin - 画板管理                                           │
├─────────────────────────────────────────────────────────────┤
│  📁 我的小屋 (5/300)                             [▼ 折叠]   │
│    ├─ 画板 0  [文字]  "今日待办..."                         │
│    ├─ 画板 1  [图片]  https://...                           │
│    ├─ 画板 2  [Canvas] drawStar                             │
│    └─ ...                                                   │
│                                                             │
│  📁 考研记忆馆 (3/300)                           [▶ 折叠]   │
│  📁 历史展览 (7/300)                             [▶ 折叠]   │
└─────────────────────────────────────────────────────────────┘

功能：
  ✅ 按建筑分组展示（可折叠）
  ✅ 编辑画板内容
  ✅ 切换画板类型
  ✅ 预览效果
```

### 6. V1.0 不做的事

```
❌ 用户系统（单用户）
❌ 笔记表（画板本身就是笔记）
❌ 笔记↔画板关联（V3.0 的事）
❌ 3D 点选编辑（V2.0 的事）
❌ Obsidian 插件（V? 的事）
❌ 复杂搜索/标签
```

### 7. 待决策问题

```
明天需要深度思考：

1. 数据存储最终形态
   - 一直用数据库？
   - 还是未来迁移到文件存储（兼容 Obsidian）？
   - 混合方案？

2. admin 的「笔记」功能
   - V1.0 不做笔记表，但未来要做吗？
   - 如果做，admin 会变成「主笔记软件」吗？
   - 还是画板就是笔记？

3. 双向互选
   - admin 笔记 ↔ 三维画板 的关联
   - V3.0 才需要？还是 V1.0 就要做？

4. 3D 点选
   - 在三维里点击画板 → 弹出编辑框
   - V2.0 第一优先级？
```

### 8. 开发原则

```
对数据结构谨慎（迁移成本高）
对功能顺势而为（边做边调整）

谨慎：
  - 数据库表结构
  - 建筑命名规则（全局唯一）
  - 画板 ID 生成规则

顺势：
  - admin 界面细节
  - 要不要 3D 点选
  - 画板编辑器功能
```
